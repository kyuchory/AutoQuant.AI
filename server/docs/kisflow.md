# 📑 [기술 문서] 한국투자증권(KIS) Open API 인프라 및 인증 플로우 설계

## 1. 기반 핵심 정보 (System Context)

- **이용 주체 구분:** 개인/일반법인 (`Client Credentials` 방식 - 사용자 개입 없는 2-legged 시스템)
- **인증 프로토콜:** OAuth 2.0
- **발급 필수 키셋 (Key Set):** `APP Key`, `APP Secret` (모의투자 계좌: `50197569`)
- **핵심 제약 조건:** * 접근 토큰(`Access Token`)의 유효기간은 **24시간**이며, 안전성을 위해 **6시간마다 갱신** 필요.
    - 토큰이 만료되면 모든 REST API 호출이 차단되므로 백그라운드 스케줄러를 통한 주기적 재발급 필수.

## 2. [플로우 1] Access Token 및 웹소켓 연결 키 발급 흐름 (REST)

애플리케이션이 구동되거나 토큰 만료 주기(6시간)에 도달했을 때 백엔드 내부에서 비동기적으로 일어나는 인증 파이프라인입니다.

```
[ 우리 Spring Boot 서버 ]                           [ 한국투자증권 API 서버 ]
       │                                                      │
       │ 1. 토큰 발급 요청 (POST /oauth2/tokenP)               │
       ├─────────────────────────────────────────────────────>│
       │    (Payload: APP Key, APP Secret, grant_type)       │
       │                                                      │
       │ 2. Access Token 발급 응답 (유효기간 24시간)            │
       │<─────────────────────────────────────────────────────┤
       │    (응답: access_token, token_type)                  │
       │                                                      │
       │ 3. [선택] 웹소켓 접속키 발급 요청 (POST /oauth2/Approval)│
       ├─────────────────────────────────────────────────────>│
       │    (Payload: APP Key, APP Secret)                    │
       │                                                      │
       │ 4. 웹소켓용 인가키 발급 응답                          │
       │<─────────────────────────────────────────────────────┤
       │    (응답: approval_key)                              │
       │                                                      │
▼ 메모리 및 Redis에 Token/Key 캐싱 완료                                   │
```

## 3. [플로우 2] 실시간 시세 수집 및 캐싱 흐름 (WebSocket)

v2 워크플로우 설계에 맞추어, REST 폴링 방식을 탈피하고 지속적인 양방향 커넥션을 유지하며 Redis에 데이터를 적재하는 동적 구독 흐름입니다.

```
[ DB (MySQL) ]     [ Spring 부트 시세 클라이언트 ]           [ KIS 웹소켓 서버 ]      [ Redis ]
      │                         │                              │                │
      │ 1. 감시 종목 조회         │                              │                │
      │    (is_monitored=TRUE)  │                              │                │
      ├────────────────────────>│                              │                │
      │    (대장주 10종 코드)      │                              │                │
      │                         │ 2. 웹소켓 핸드셰이크 요청       │                │
      │                         │    (with approval_key)       │                │
      │                         ├─────────────────────────────>│                │
      │                         │                              │                │
      │                         │ 3. 실시간 체결가 구독 등록    │                │
      │                         │    (Tr_Id: H0STCNT0, 종목코드) │                │
      │                         ├─────────────────────────────>│                │
      │                         │                              │                │
      │                         │ 4. [LOOP] 실시간 시세 Push   │                │
      │                         │<─────────────────────────────┤                │
      │                         │    (종목별 체결가 데이터 수신)  │                │
      │                         │                              │                │
      │                         │ 5. 초고속 데이터 적재          │                │
      │                         ├──────────────────────────────font────────────>│
      │                         │                              │ (stock:price:) │
```

## 4. [호출 방식 명확화] 구조적 역할 분담

백엔드 아키텍처 대시보드 내에서 KIS API를 찌르는 두 가지 통신 방식의 타겟 레이어입니다.

| 통신 프로토콜 | 주 사용 목적 (우리 프로젝트 기준) | 빈도 및 특징 |
| --- | --- | --- |
| **WebSocket** | 국내 주식 대장주 10종목의 **실시간 체결가 및 호가 수집** | **장중 지속(Push)**, 끊김 시 재연결 로직 세트로 묶음 |
| **REST API** | ① 최초 구동 시 access_token/approval_key 인증
② 장 시작 직후 종목별 최초 스냅샷 조회
③ 조건 매칭 시 **모의투자 매수/매도 주문 실행**
④ 유저 계좌 잔고 및 수익률 동기화 조회 | **단발성(Request-Response)**, 호출 제한(Rate Limit) 방어선 필요 |
